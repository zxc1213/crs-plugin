# 需求文档迁移命令（历史迁移工具）

> **⚠️ 重要提示**：这是一个**一次性迁移工具**，仅用于从 v0.6.0 之前版本升级。
>
> 如果你是新用户（v0.6.0+），所有需求已自动使用新格式，**无需使用此命令**。

## 与 /req:unify 的区别

| 特性         | /req:migrate                    | /req:unify               |
| ------------ | ------------------------------- | ------------------------ |
| **备份**     | ✅ 自动备份到 `.backup/`        | ❌ 无备份                |
| **回滚**     | ✅ 支持回滚到迁移前             | ❌ 无回滚                |
| **适用场景** | ⭐ **推荐首次迁移**（重要需求） | 快速统一（可丢弃的需求） |
| **安全性**   | 高（有保险）                    | 低（无保险）             |

**推荐**：首次迁移旧格式需求时使用 `/req:migrate`，更安全。

## 背景说明

### 为什么要迁移？

v0.6.0 之前，每个需求使用 5 个独立文件：

```
.requireations/FEAT-20260514-001/
├── design.md           # 设计文档
├── test-plan.md        # 测试计划
├── plan.md             # 实施计划
├── analysis-report.md  # 分析报告
└── CHANGELOG.md        # 变更日志
```

**问题**：维护成本高、信息冗余、容易不一致

### v0.6.0+ 新格式

所有信息统一到 1 个文件：

```
.requireations/FEAT-20260514-001/
└── spec.md             # 统一规格文档
```

## 用法

```bash
# 迁移单个需求（推荐）
/req:migrate FEAT-20260514-001

# 批量迁移所有需求
/req:migrate --all

# 预览模式（不实际修改）
/req:migrate FEAT-20260514-001 --dry-run

# 交互式迁移（逐步确认）
/req:migrate FEAT-20260514-001 --interactive

# 查看迁移状态
/req:migrate --status
```

## 选项说明

| 选项            | 说明                                     |
| --------------- | ---------------------------------------- |
| `req-id`        | 需求 ID（如：FEAT-20260514-001）         |
| `--all`         | 批量迁移所有旧格式需求                   |
| `--dry-run`     | 预览模式，显示将要执行的操作但不实际修改 |
| `--interactive` | 交互式迁移，每个步骤都需确认             |
| `--status`      | 查看迁移状态和统计                       |

## 安全特性

### ✅ 自动备份

迁移前自动创建 `.backup/` 目录，保存所有旧文件的完整副本：

```
.requirements/.backup/FEAT-20260514-001/
├── design.md
├── test-plan.md
├── plan.md
├── analysis-report.md
└── CHANGELOG.md
```

### ✅ 验证检查

迁移后自动验证：

- 内容完整性（所有信息都已迁移）
- 格式正确性（spec.md 格式符合规范）
- 元数据一致（ID、类型、状态等）

### ✅ 回滚机制

如果迁移后发现问题，可以回滚：

```bash
# 回滚特定需求
/req:migrate --rollback FEAT-20260514-001

# 回滚所有迁移
/req:migrate --rollback --all
```

## 注意事项

### ⚠️ 迁移前

1. **确认需求状态**：最好在需求非活跃状态时迁移
2. **使用 --interactive**：重要需求建议使用交互式模式
3. **备份整个项目**：虽然命令会自动备份，但额外的项目级备份更安全

### ✅ 迁移后

1. **验证 spec.md 内容**：检查所有信息是否正确迁移
2. **运行质量检查**：`/req:quality <req-id>` 确保文档质量
3. **测试需求流程**：确认需求管理工作流正常

### 🔄 回滚场景

如果出现以下情况，考虑回滚：

- spec.md 格式错误
- 重要信息丢失
- 元数据不一致

## 常见问题

### Q: 我需要运行这个命令吗？

**A**: 如果你：

- ✅ 是新用户（v0.6.0+ 安装）→ **不需要**
- ✅ 所有需求都只有 `spec.md` → **不需要**
- ⚠️ 从旧版本升级且需求有多个 .md 文件 → **需要**

### Q: /req:migrate 和 /req:unify 有什么区别？

**A**: 两个命令都能完成格式统一，但：

- **使用 /req:migrate**：重要需求、首次迁移、需要保险
- **使用 /req:unify**：不重要需求、快速统一、可丢弃

### Q: 迁移会丢失数据吗？

**A**: 不会。迁移只是合并文件，所有信息都会保留到 spec.md。而且有备份，可以随时回滚。

### Q: 备份文件会占用多少空间？

**A**: 备份文件与原文件大小相同。如果确认迁移无误，可以手动删除 `.backup/` 目录节省空间。

### Q: 可以只迁移部分需求吗？

**A**: 可以。使用 `/req:migrate <req-id>` 逐个迁移，或者多次运行命令迁移不同需求。
