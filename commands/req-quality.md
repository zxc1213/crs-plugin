# 需求质量检查命令

在需求管理的各个关键阶段执行质量门禁检查，确保每个阶段的工作质量符合标准。

## 用法

```bash
/req:quality <req-id> [--gate <1-4>] [--auto]
```

## 选项

- `req-id` - 需求 ID（如：REQ-20260514-001）
- `--gate <1-4>` - 指定检查点（默认：自动推断）
  - `1` - 设计完成检查
  - `2` - 测试策略完成检查
  - `3` - 实施计划完成检查
  - `4` - 变更完成检查
- `--auto` - 自动模式，跳过手动确认项

## 检查点

### Gate 1: 设计完成检查

**触发时机**：设计文档完成后

**自动检查项**：

- ✓ 无占位符（TBD/TODO）
- ✓ 无矛盾内容
- ✓ 决策理由完整
- ✓ 架构完整性
- ✓ 错误处理覆盖

**手动确认项**：

- 所有决策都有明确理由
- 架构图清晰完整
- 错误处理覆盖主要场景
- 接口定义明确
- 数据流完整

**通过条件**：所有自动检查通过 + 至少 80% 手动确认项完成

### Gate 2: 测试策略完成检查

**触发时机**：测试计划生成后

**自动检查项**：

- ✓ 测试类型覆盖（≥3种）
- ✓ 关键路径测试
- ✓ 性能指标定义
- ✓ 安全风险识别

**手动确认项**：

- 测试场景覆盖业务需求
- 测试数据准备充分
- 验收标准明确
- 测试环境配置清晰

### Gate 3: 实施计划完成检查

**触发时机**：实施计划生成后

**自动检查项**：

- ✓ 任务可执行性
- ✓ 依赖关系清晰
- ✓ 时间估算合理
- ✓ 风险预案完整

**手动确认项**：

- 任务分解粒度合适
- 资源分配合理
- 里程碑设置明确
- 团队成员确认

### Gate 4: 变更完成检查

**触发时机**：需求变更处理完

**自动检查项**：

- ✓ 影响评估完整
- ✓ 文档已更新
- ✓ 测试已调整
- ✓ 干系人已通知

**手动确认项**：

- 变更理由充分
- 影响范围准确
- 实施方案可行
- 回滚计划准备

## 重要说明

**DO:**

- ✅ 在每个关键阶段完成后执行质量检查
- ✅ 仔细审查自动检查报告
- ✅ 完成手动确认项后再进入下一阶段
- ✅ 记录质量问题到需求文档

**DON'T:**

- ❌ 跳过质量门禁检查
- ❌ 忽略阻塞性问题
- ❌ 在检查未通过时进入下一阶段
- ❌ 伪造手动确认项

## 相关命令
