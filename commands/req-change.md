# 需求变更处理命令

在任务执行过程中处理需求变更，评估影响并调整执行计划。

## 什么是需求变更？

在需求执行过程中，可能会出现以下情况：

- 💡 **新想法**：团队想到了更好的实现方式
- 📋 **需求调整**：产品/用户提出修改要求
- 🐛 **发现新问题**：实施过程中发现未预见的问题
- ⏰ **外部变化**：市场、技术环境发生变化

**需求变更处理命令**帮助你：

1. 评估变更影响（有多大？）
2. 调整执行计划（怎么办？）
3. 记录变更历史（为什么变？）

## 用法

```bash
# 处理需求变更
/req:change FEAT-20260514-001 --reason "用户要求增加头像上传功能"

# 显示影响分析
/req:change FEAT-20260514-001 --reason "..." --impact

# 指定变更类型
/req:change FEAT-20260514-001 --type major --reason "架构调整"
```

## 选项说明

| 选项              | 说明                             |
| ----------------- | -------------------------------- |
| `req-id`          | 需求 ID（如：FEAT-20260514-001） |
| `--reason <理由>` | 变更理由（必填，说明为什么变更） |
| `--type <类型>`   | 指定变更类型：small/medium/large |
| `--impact`        | 显示详细影响分析                 |

## 变更分类

系统会根据变更的**影响范围**和**实施成本**自动分类：

### 小调整 (Small)

**特征**：

- 文案、样式微调
- 配置参数修改
- 预计耗时 < 5分钟

**处理方式**：

- ✅ 直接执行
- ✅ 无需重新规划
- ✅ 记录到变更日志

**示例**：

```bash
/req:change FEAT-20260514-001 --reason "按钮文案从'提交'改为'保存'"
```

**输出**：

```markdown
📝 需求变更 - FEAT-20260514-001

🔍 变更分类: 小调整
✅ 变更完成，继续执行
```

### 中等变更 (Medium)

**特征**：

- 影响 1-2 个任务
- 需要调整实施计划
- 预计耗时 < 1小时

**处理方式**：

- ⚠️ 快速审查后调整
- ✅ 更新相关任务
- ✅ 记录影响分析

**示例**：

```bash
/req:change FEAT-20260514-001 --reason "增加密码强度验证" --impact
```

**输出**：

```markdown
📝 需求变更 - FEAT-20260514-001

🔍 变更分类: 中等变更

⚠️ 影响评估:
影响任务: 2个

- FEAT-20260514-001-T03: 前端表单验证
- FEAT-20260514-001-T05: 后端验证逻辑

预估耗时: 30分钟

建议: 更新任务 FEAT-20260514-001-T03 和 FEAT-20260514-001-T05

✅ 已调整执行计划
📝 已记录变更历史
```

### 重大变更 (Large)

**特征**：

- 改变架构或核心流程
- 影响多个任务（>3个）
- 预计耗时 > 1小时

**处理方式**：

- 🚨 完整分析后重新规划
- ✅ 更新所有相关任务
- ✅ 重新评估优先级
- ✅ 可能需要重新分析需求

**示例**：

```bash
/req:change FEAT-20260514-001 --type major --reason "改用实时通信替代轮询"
```

**输出**：

```markdown
🚨 重大需求变更 - FEAT-20260514-001

⚠️ 影响评估:
架构变更: 是
影响任务: 8个 (全部)

受影响的任务:

- FEAT-20260514-001-T01: 需求分析 (需重新)
- FEAT-20260514-001-T02: 架构设计 (需重新)
- FEAT-20260514-001-T03: 后端API (重大调整)
- FEAT-20260514-001-T04: 前端实现 (重大调整)
- ...

预估耗时: 4-6小时
风险评估: 高

建议:

1. 重新分析需求 (req-brainstorm)
2. 更新架构设计
3. 重新制定实施计划
4. 更新测试策略

❓ 是否继续? [Y/n]
```

## 使用场景

### 场景1：产品经理要求调整

```bash
# PM说：登录页面需要添加验证码
/req:change FEAT-20260514-001 --reason "PM要求添加验证码功能" --impact
```

系统会：

1. 评估影响（中等变更）
2. 识别受影响的任务
3. 建议调整实施计划
4. 记录变更历史

### 场景2：发现更好的实现方式

```bash
# 开发中发现：用WebSocket比轮询更好
/req:change FEAT-20260514-001 --type major --reason "WebSocket实时性更好，架构更优"
```

系统会：

1. 识别为重大变更
2. 显示完整影响分析
3. 建议重新分析需求
4. 等待确认后执行

### 场景3：用户反馈调整

```bash
# 用户说：按钮位置不合理
/req:change FEAT-20260514-001 --reason "用户反馈：按钮位置不直观，需要调整"
```

系统会：

1. 识别为小调整
2. 直接执行并记录
3. 继续执行任务

## 变更历史记录

所有变更都会记录到需求的 `spec.md` 变更历史部分：

```markdown
## 变更历史

| 日期       | 变更内容          | 影响分析         | 变更类型 |
| ---------- | ----------------- | ---------------- | -------- |
| 2026-05-14 | 创建需求          | -                | -        |
| 2026-05-15 | 添加密码强度验证  | 影响T03、T05任务 | 中等变更 |
| 2026-05-16 | 改用WebSocket架构 | 重新规划全部任务 | 重大变更 |
```

## 重要说明

**DO:**

- ✅ 及时记录需求变更（不要事后补）
- ✅ 详细说明变更理由（为什么变？）
- ✅ 评估变更影响（有多大？）
- ✅ 重大变更前与团队讨论

**DON'T:**

- ❌ 频繁变更（说明需求分析不充分）
- ❌ 不记录变更（丢失决策历史）
- ❌ 忽略影响评估（可能导致延期）
- ❌ 重大变更不经讨论直接执行

## 最佳实践

### 1. 变更前思考

```
这个变更是必须的吗？
- 如果不变更会怎样？
- 有没有更好的替代方案？
- 现在是变更的最佳时机吗？
```

### 2. 小批量变更

```
尽量将变更拆分为小批次：
- 先变必须的
- 后变可选的
- 避免一次大变更
```

### 3. 保持文档同步

```
变更后立即更新：
- spec.md（主文档）
- 任务清单
- 测试用例
```
