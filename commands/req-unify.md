# 需求文档格式统一（历史迁移工具）

> **⚠️ 重要提示**：这是一个**一次性迁移工具**，仅用于从 v0.6.0 之前版本升级。
>
> 如果你是新用户（v0.6.0+），所有需求已自动使用新格式，**无需使用此命令**。

## 背景说明

### v0.6.0 之前（旧格式）

每个需求使用 5 个独立文件存储信息：

```
.requireations/FEAT-20260514-001/
├── design.md           # 设计文档
├── test-plan.md        # 测试计划
├── plan.md             # 实施计划
├── analysis-report.md  # 分析报告
└── CHANGELOG.md        # 变更日志
```

**问题**：

- 维护成本高（5个文件）
- 信息冗余（同一信息在多个文件重复）
- 容易不一致（修改一处忘了更新另一处）
- 跳转频繁（查看信息需要打开多个文件）

### v0.6.0+（新格式）

所有信息统一到 1 个文件：

```
.requireations/FEAT-20260514-001/
└── spec.md             # 统一规格文档（包含所有信息）
```

## 用法

```bash
# 迁移单个需求
/req:unify FEAT-20260514-001

# 迁移所有需求（批量操作）
/req:unify --all

# 预览模式（不实际修改）
/req:unify FEAT-20260514-001 --dry-run

# 强制覆盖已存在的 spec.md
/req:unify FEAT-20260514-001 --force
```

## 选项说明

| 选项        | 说明                                     |
| ----------- | ---------------------------------------- |
| `req-id`    | 需求 ID（如：FEAT-20260514-001）         |
| `--all`     | 批量迁移所有旧格式需求                   |
| `--dry-run` | 预览模式，显示将要执行的操作但不实际修改 |
| `--force`   | 强制覆盖已存在的 spec.md（谨慎使用）     |


## 注意事项

### ⚠️ 迁移前建议

1. **备份重要需求**：对于重要的需求，先备份整个需求目录
2. **使用 --dry-run 预览**：先运行预览模式确认操作正确
3. **逐个迁移**：建议逐个迁移而不是批量 --all，便于检查

### ✅ 迁移后验证

检查 `spec.md` 是否包含所有必要信息：

- 元数据完整
- 设计方案已迁移
- 测试计划已迁移
- 变更历史已保留

### 🔄 不可逆操作

**迁移会删除旧文件**，如果迁移后发现有问题：

- 可以从 git 历史恢复旧格式
- 或手动从 `spec.md` 中拆分出需要的部分

## 常见问题

### Q: 我需要运行这个命令吗？

**A**: 如果你：

- ✅ 是新用户（v0.6.0+ 安装）→ **不需要**
- ✅ 所有需求都在 `.requirements/` 目录下且只有 `spec.md` → **不需要**
- ⚠️ 从旧版本升级且需求目录有多个 .md 文件 → **需要**

### Q: 迁移会丢失数据吗？

**A**: 不会。迁移只是将 5 个文件的内容合并到 1 个文件，所有信息都会保留。

### Q: 可以恢复到旧格式吗？

**A**: 可以，通过 git 历史恢复。但新格式更简洁高效，建议保持新格式。
