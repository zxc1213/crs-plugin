# 需求管理系统命令

智能需求管理系统，为 Claude Code 提供从需求捕获到测试完成的全流程管理能力。

**核心特性**：

- 集成 `req-brainstorm` 深度分析模式，确保每个需求都经过完整的探索和审查
- `req-manager` 统一入口，智能路由到最优流程 ⭐ 新增
- `req-priority` 科学评估优先级，优化资源分配 ⭐ 新增
- `req-quality` 质量门禁系统，确保交付质量 ⭐ 新增
- `req-unify` 统一文档结构，降低维护成本 ⭐ 新增

## 用法

```bash
/req [选项] <描述>
```

## 选项

### 类型选项

- `--feature`, `-f` : 新功能（默认）
- `--bug`, `-b` : Bug 修复
- `--question`, `-q` : 技术问题
- `--adjust`, `-a` : 需求调整
- `--refactor`, `-r` : 重构

### 分析选项

- `--quick` : 快速模式，跳过深度分析
- `--deep`, `-d` : 深度分析模式（默认，使用 req-brainstorm）

### 执行选项

- `--auto` : 全自动模式
- `--conservative` : 保守模式

### 查询选项

- `--list` : 列出所有需求
- `--active` : 当前活跃需求
- `--status <id>` : 需求状态
- `--dashboard` : 显示仪表板

## 命令行为

### 查询命令

**当用户使用查询选项时**：

1. 运行对应的查询脚本
2. 格式化并显示结果
3. 不创建任何需求文件

**`--dashboard`**：

- 运行 `node .claude/scripts/requirement-manager/index.js --dashboard`
- 显示需求统计、活跃需求、最近需求列表

**`--list`**：

- 运行 `node .claude/scripts/requirement-manager/index.js --list`
- 列出所有需求及其状态

**`--active`**：

- 运行 `node .claude/scripts/requirement-manager/index.js --active`
- 显示当前活跃（in_progress）的需求

### 创建需求（集成深度分析）

**核心流程**：当用户提供描述时，自动启动 `req-brainstorm` 深度分析流程。

```
需求输入
  ↓
[阶段1] 需求解析
  → 识别需求类型
  → 检测敏感信息
  → 查找相似需求
  ↓
[阶段2] 深度分析（req-brainstorm）
  → 问题探索（Brainstorming）
  → 方案审查（Grill-me）
  → 设计展示
  → 最终审查
  ↓
[阶段3] 需求创建
  → 生成唯一 ID
  → 创建需求目录
  → 生成设计文档
  → 创建 meta.yaml
  ↓
[阶段4] 执行计划
  → 调用 writing-plans
  → 生成实现计划
```

#### 阶段1: 需求解析

**自动执行**：

1. 解析需求类型（从选项或描述推断）
2. **安全检查**：
   - 调用 SecurityFilter 检测敏感信息
   - 发现敏感信息时显示警告并提供脱敏建议
   - 等待用户确认后继续

3. **相似度检测**：
   - 调用 SimilarityDetector 查找相似需求
   - 显示相似需求列表
   - 询问是否继续创建新需求

#### 阶段2: 深度分析（req-brainstorm）

**除非使用 `--quick` 选项，否则自动启动**：

**触发 `req-brainstorm` skill**，执行四阶段分析：

1. **问题探索**（Brainstorming 主导）
   - 探索项目上下文
   - 一次一个问题地理解需求
   - 提出 2-3 个备选方案

2. **方案审查**（Grill-me 主导）
   - 逐分支深度提问
   - 遍历决策树
   - 挑战假设，提供推荐答案

3. **设计展示**（Brainstorming 主导）
   - 分段展示设计
   - 逐段确认

4. **最终审查**（Grill-me 主导）
   - 完整性检查
   - 生成决策摘要

**输出**：完整的设计文档

#### 阶段2.5: 优先级评估 ⭐ 新增

**分析完成后自动启动**：

**触发 `req-priority` skill**，执行多维度评估：

1. **业务价值评估**（权重40%）
   - 核心业务功能 → 重要功能 → 改进性功能 → 可选功能

2. **紧急程度评估**（权重30%）
   - 阻塞性问题 → 紧急修复 → 重要但不阻塞 → 一般问题

3. **依赖关系分析**（权重15%）
   - 多个高优先级依赖 → 重要需求前置 → 有一些依赖 → 依赖较少

4. **实现成本评估**（权重10%）
   - 非常简单 → 简单 → 中等 → 复杂 → 非常复杂

5. **风险评估**（权重5%）
   - 低风险 → 较低风险 → 中等风险 → 较高风险 → 极高风险

**输出**：

- 综合评分（0-10分）
- 优先级等级（P0/P1/P2/P3/P4）
- ROI 评估
- 详细评分说明

#### 阶段2.75: 质量门禁检查（Gate 1）⭐ 新增

**优先级评估完成后自动启动**：

**触发 `req-quality` skill**，检查设计质量：

**自动检查项**：

- ✓ 无占位符（TBD/TODO）
- ✓ 无矛盾内容
- ✓ 决策理由完整
- ✓ 架构完整性
- ✓ 错误处理覆盖

**手动确认项**：

- 所有决策都有明确理由
- 架构图清晰完整
- 错误处理覆盖主要场景
- 接口定义明确
- 数据流完整

**通过条件**：所有自动检查通过 + 至少 80% 手动确认项完成

**输出**：质量检查报告（通过/失败 + 改进建议）

#### 阶段3: 需求创建

**质量检查通过后自动执行**：

1. 生成唯一的需求 ID（格式：REQ-YYYYMMDD-XXX）
2. 在 `.requirements/<type>/REQ-XXX/` 创建目录
3. 创建以下文件（新格式）⭐ 更新：

   ```
   .requirements/<type>/REQ-XXX/
   ├── meta.yaml              # 需求元数据（含优先级信息）⭐ 新增
   ├── spec.md                # 统一主文档 ⭐ 新格式
   └── test-cases.md          # 详细测试用例（可选）⭐ 新格式
   ```

4. **meta.yaml 结构**⭐ 更新：

   ```yaml
   id: REQ-20260513-001
   type: feature
   title: 需求标题
   description: 需求描述
   status: planning
   created_at: 2026-05-13T10:00:00Z
   updated_at: 2026-05-13T10:00:00Z
   analysis_method: req-brainstorm
   priority: ⭐ 新增
     level: P0  # 优先级等级
     score: 8.6  # 综合评分
     breakdown:  # 详细评分
       business_value: 9
       urgency: 8
       dependencies: 8
       effort: 8
       risk: 9
     rationale: "核心业务功能，多模块依赖，技术成熟"
     estimated_effort: "2-3天"
     roi: "高"
   design_decisions:
     - decision: "决策1"
       rationale: "理由"
     - decision: "决策2"
       rationale: "理由"
   open_questions: []
   dependencies: []
   related_requirements: []
   ```

5. **spec.md 结构**⭐ 新增：

   ```markdown

   - **ID**: REQ-20260513-001
   - **类型**: feature
   - **状态**: planning

   - **创建时间**: 2026-05-13

   ## 需求分析

   ### 背景与目标

   ### 用户故事

   ### 验收标准

   ### 开放问题

   ## 设计方案

   ### 系统架构

   ### 核心组件

   ### 数据流设计

   ### 接口定义

   ### 技术选型

   ### 决策记录

   ### 错误处理

   ### 安全考虑

   ## 测试策略

   ### 测试范围

   ### 测试类型

   ### 验收标准

   ### 性能指标

   ## 实施计划

   ### 任务分解

   ```

6. 更新系统指标

#### 阶段4: 执行计划

**自动调用 `writing-plans` skill**：

- 基于设计文档生成详细实现计划
- 保存为 `plan.md`
- 返回下一步操作建议

## 执行模式

### 默认模式（半自动 + 深度分析）

- 启动 req-brainstorm 深度分析
- 生成设计文档和实现计划
- 返回下一步操作建议

### 快速模式（`--quick`）

- 跳过深度分析
- 直接创建需求文件
- 生成基础执行计划

### 全自动模式（`--auto`）

- 深度分析 + 自动实现
- 自动调用相应 skill
- 创建完整的规范文档

### 保守模式（`--conservative`）

- 深度分析 + 每步确认
- 需要用户确认后继续

## 安全检查

系统会自动检测并警告：

- 敏感信息（密钥、密码、个人信息）
- API 密钥泄露
- IP 地址暴露

检测到敏感信息时会：

1. 显示警告
2. 提供脱敏建议
3. 等待用户确认后继续

## 集成说明

此命令集成了以下功能模块：

**核心模块**：

- **Processor**: 需求创建和状态管理
- **SecurityFilter**: 敏感信息检测和脱敏
- **SimilarityDetector**: 相似需求检测
- **Scheduler**: 执行计划生成和 Skill 调用
- **Dashboard**: 需求可视化和统计

**Skills 集成**⭐ 更新：

- **req-manager**: 统一入口，智能路由（自动触发）⭐ 新增
- **req-brainstorm**: 深度需求分析（默认启用）
- **req-priority**: 优先级科学评估（自动触发）⭐ 新增
- **req-quality**: 质量门禁系统（自动触发）⭐ 新增
- **req-unify**: 文档结构统一（自动使用新格式）⭐ 新增
- **req-test-plan**: 测试策略生成（自动触发）
- **writing-plans**: 实现计划生成
- **类型特定 skills**: 根据需求类型自动调用

**需要 Node.js 环境支持**（用于查询命令）。
