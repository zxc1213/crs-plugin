# 需求验证检查命令

在需求管理的各个关键阶段提供系统性的验证检查清单，确保每个阶段的工作质量和完整性。

## 与 /req:quality 的区别

| 特性         | /req:quality         | /req:verify          |
| ------------ | -------------------- | -------------------- |
| **检查方式** | 🤖 自动检查为主      | ✋ 手动检查清单      |
| **触发时机** | 在关键阶段**完成时** | 在关键阶段**开始前** |
| **检查重点** | 文档质量、完整性     | 验证标准、验收条件   |
| **通过标准** | 自动评分（≥80分）    | 人工确认（逐项勾选） |
| **适用场景** | 质量门禁             | 最终验收             |

**简而言之**：

- **/req:quality** = "文档写得好不好？"（质量检查）
- **/req:verify** = "功能做对了没有？"（验收检查）

## 用法

```bash
# 设计阶段验证
/req:verify design FEAT-20260514-001

# 实现阶段验证
/req:verify implementation FEAT-20260514-001

# 部署阶段验证
/req:verify deployment FEAT-20260514-001
```

## 检查点类型

### Checkpoint 1: 设计验证 (design)

**触发时机**：需求设计完成后，**开始实现前**

**检查清单**：

```markdown
## 设计验证清单 - FEAT-20260514-001

### 需求理解

- [ ] 完全理解用户需求
- [ ] 明确业务目标和价值
- [ ] 识别所有干系人

### 技术方案

- [ ] 架构设计完整
- [ ] 技术选型合理
- [ ] 接口定义清晰
- [ ] 数据流设计完整

### 实施可行性

- [ ] 任务可执行
- [ ] 依赖关系明确
- [ ] 时间估算合理
- [ ] 风险已识别

### 质量标准

- [ ] 验收标准明确
- [ ] 测试策略完整
- [ ] 性能指标定义
```

### Checkpoint 2: 实现验证 (implementation)

**触发时机**：代码实现完成后，**提交测试前**

**检查清单**：

```markdown
## 实现验证清单 - FEAT-20260514-001

### 代码质量

- [ ] 代码符合规范
- [ ] 注释充分清晰
- [ ] 无明显技术债
- [ ] 错误处理完整

### 功能实现

- [ ] 所有功能点已实现
- [ ] 符合设计文档
- [ ] 边界情况处理
- [ ] 用户体验良好

### 测试覆盖

- [ ] 单元测试通过
- [ ] 集成测试通过
- [ ] 边界测试覆盖
- [ ] 性能测试通过
```

### Checkpoint 3: 部署验证 (deployment)

**触发时机**：准备部署到生产环境前

**检查清单**：

```markdown
## 部署验证清单 - FEAT-20260514-001

### 功能验证

- [ ] 所有功能正常
- [ ] 无回归问题
- [ ] 性能符合预期
- [ ] 用户体验良好

### 安全检查

- [ ] 无安全漏洞
- [ ] 权限控制正确
- [ ] 数据安全保护
- [ ] 日志审计完整

### 运维准备

- [ ] 监控配置完成
- [ ] 告警规则设置
- [ ] 回滚方案准备
- [ ] 文档更新完整
```

## 验证状态

| 状态        | 条件           | 建议                 |
| ----------- | -------------- | -------------------- |
| ✅ 通过     | 所有检查项完成 | 可以进入下一阶段     |
| ⚠️ 部分通过 | 通过率 ≥ 80%   | 完成关键缺失项后继续 |
| ❌ 未通过   | 通过率 < 80%   | 需要全面整改         |

## 何时使用

### ✅ 应该使用

1. **设计完成后**：在开始编码前，验证设计是否完整可行
2. **实现完成后**：在提交测试前，验证功能是否正确实现
3. **部署前**：在上生产前，验证所有准备工作是否完成

### ❌ 不需要使用

1. **快速原型**：如果是探索性原型，可以跳过
2. **小修改**：如果是文案调整、样式修改等小改动
3. **紧急修复**：如果是紧急Bug修复，可以事后补验证

## 工作流示例

```bash
# 1. 创建需求（自动质量检查）
/req 添加用户登录功能

# 2. 设计完成后，验证设计
/req:verify design FEAT-20260514-001

# 3. 实现完成后，验证实现
/req:verify implementation FEAT-20260514-001

# 4. 部署前，验证部署准备
/req:verify deployment FEAT-20260514-001

# 5. 全部通过后，部署上线
```

## 重要说明

**DO:**

- ✅ 在每个关键阶段后执行验证检查
- ✅ 仔细逐项勾选检查清单
- ✅ 对未完成项制定改进计划
- ✅ 记录验证结果到需求文档

**DON'T:**

- ❌ 跳过验证检查直接进入下一阶段
- ❌ 伪造检查项（假装已完成）
- ❌ 忽略关键缺失项
- ❌ 在验证未通过时强行推进
